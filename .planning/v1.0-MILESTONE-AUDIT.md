### High-Severity Integration Gaps

#### GAP-05: 4 CSS Custom Properties Undefined in Phase 9/10 Components
**Severity:** High
**Affects:** COMM-01, COMM-02, COMM-03, COMM-04, COMM-05, COMM-06, SOCIAL-04, UX-01, UX-04 (visual rendering)
**File:** `src/lib/styles/theme.css`

Phase 8 established the canonical CSS variable naming: `--bg-base`, `--bg-surface`, `--bg-elevated`, `--border-default`, `--link-color`. Phases 9 and 10 components were authored using a different (undefined) naming convention: `--bg-primary`, `--bg-tertiary`, `--border`, `--accent`.

**67 usages of undefined variables across Phase 9/10 components:**

| Variable | Uses | Visible consequence |
|---|---|---|
| `--accent` | 19 | Transparent color on buttons, active states, constellation nodes |
| `--bg-primary` | 8 | Transparent background on AiGatePrompt, RoomCreator, RoomDirectory, SessionCreator |
| `--bg-tertiary` | 11 | Transparent background on ModerationQueue, RoomCreator, RoomDirectory, SessionCreator, CollectionShelf, TasteFingerprint |
| `--border` | 29 | Invisible borders on chat components, AvatarEditor, CollectionShelf, TasteFingerprint, artist/release pages |

Affected buttons (transparent = invisible to user): "Create Room," "Start Party," "Open Settings" (AI gate), all chat component borders, "Save to Shelf" confirmation state, profile active tab.

**Fix (one-line to 4-line):** Add to `src/lib/styles/theme.css`:
```css
--bg-primary: var(--bg-base);
--bg-tertiary: var(--bg-elevated);
--border: var(--border-default);
--accent: var(--link-color);
```

#### GAP-06: DM Conversation List Missing — COMM-04 Functionally Inaccessible
**Severity:** High
**Affects:** COMM-04 (Private DM system)
**Files:** `src/lib/components/chat/ChatOverlay.svelte`, `src/lib/components/chat/ChatPanel.svelte`

The NIP-17 DM backend is fully implemented and verified: `dms.svelte.ts` correctly receives, decrypts, and stores incoming DMs in `dmState.conversations`; `sendDM()` gift-wraps and publishes correctly. However, the ChatOverlay's "DMs" tab renders only a static placeholder: "Select a conversation, room, or listening party."

No component renders `dmState.conversations` as a list. No code path in the entire system sets `chatState.view = 'dm-thread'` or `chatState.activeConversationPubkey`. There is no UI to:
1. View a list of DM conversations
2. Open a specific conversation thread
3. Initiate a new DM with a pubkey

The DM system exists and works at the data/protocol layer but is completely inaccessible from the UI.

**Fix:** Add a `ConversationList.svelte` component rendered in `ChatOverlay`'s 'dms' view, iterating `dmState.conversations`, with click handlers that set `chatState.view = 'dm-thread'` and `chatState.activeConversationPubkey`. Include a "New DM" pubkey input.

### Low-Severity Integration Gaps (Not Blocking)

#### GAP-07: Nostr Taste Profile (`kind:30078`) Never Published
**Severity:** Low
**Affects:** COMM-04 (AI taste bridge quality, not core functionality)

`ai-taste-bridge.ts:81–104` fetches peer `kind:30078` events to build a musical bridge between two users' tastes. Phase 9 (Community Foundation) never publishes these events. The bridge always falls back to "Peer hasn't published a taste profile" and generates output using only the local user's context. The fallback path is functional — the bridge generates output and conversation starters, just without peer context.

**Fix (low urgency):** Add `kind:30078` publishing from the Profile page when user sets their handle, or from the taste profile save flow.

#### GAP-08: `export_play_history_to_path` Tauri Command Unimplemented (Pre-existing)
**Severity:** Low (known, non-required feature)
**Affects:** PLAYER-03 (export sub-feature only — core playback→taste pipeline is verified)

`src/lib/taste/history.ts:74` calls `invoke('export_play_history_to_path')` which is not implemented in `taste_db.rs` and not registered in `lib.rs`. The "Export JSON" button in Settings > Listening History fails silently. Documented in Phase 07.2 VERIFICATION.md as a gap. The PLAYER-03 requirement core (playback→taste signal) is fully satisfied — this is a non-required UX feature in the same settings panel.

**Fix:** Implement `export_play_history_to_path(path: String, json: String, state: ...)` in `taste_db.rs` + register in `lib.rs generate_handler!`.
