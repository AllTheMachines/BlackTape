# Phase 06.1 Plan 02: Release Page Data Layer Summary

**One-liner:** SSR + Tauri data layer for release pages — MusicBrainz browse endpoint with Cloudflare caching and server-side affiliate link construction.

## What Was Built

Two SvelteKit route files for `/artist/{slug}/release/{mbid}`:

**`+page.server.ts`** — SSR load (Cloudflare Pages):
- Guards on `platform?.env?.DB` — returns null shell for Tauri dev where server is unavailable
- Fetches MusicBrainz browse endpoint: `/ws/2/release?release-group={mbid}&inc=recordings+artist-credits+media+artist-rels`
- Cloudflare Cache API with 24hr TTL (matches existing API route pattern)
- Extracts tracklist (`Track[]`), personnel credits (`Credit[]`), release-level Bandcamp URL
- Calls `getAffiliateConfig()` + `buildBuyLinks()` server-side where env vars are available
- Exports `ReleaseDetail`, `Track`, `Credit` interfaces for use by `+page.ts`

**`+page.ts`** — Universal load (web passthrough / Tauri direct):
- Web: `isTauri() === false` — returns server data unchanged
- Tauri: fetches MusicBrainz directly (no Cloudflare cache available)
- Dynamic import of `buildBuyLinks` (NOT `getAffiliateConfig` — server-only import would fail)
- Passes null affiliate config — buy links degrade gracefully to non-coded search fallbacks
- Mirrors the pattern of `src/routes/artist/[slug]/+page.ts`
