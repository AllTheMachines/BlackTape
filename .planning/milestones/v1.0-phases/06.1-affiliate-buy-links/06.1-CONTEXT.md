# Phase 06.1: Affiliate Buy Links - Context

**Gathered:** 2026-02-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Add affiliate-coded purchase links to Mercury release pages, plus create the release pages themselves (currently missing — artist pages show release lists but there's no way to navigate into an individual release). Platforms: Bandcamp, Amazon Music, Apple Music, Beatport, Discogs Marketplace. Mercury earns passive commission when users click through and buy.

New affiliate programs, tracking dashboards, or analytics are out of scope for this phase.

</domain>

<decisions>
## Implementation Decisions

### Release Pages (new in this phase)
- Release pages must be **created as part of Phase 06.1** — they don't exist yet
- Current state: artist page shows a list of releases, no clickable through to individual release
- Release pages should be **rich**: cover art, tracklist from MusicBrainz, personnel/credits, 'Listen on' stream bar, and the new 'Buy on' buy link row
- Artist pages may also surface general buy links (researcher should determine what makes sense per-platform)

### Link Placement & Presentation
- Buy links live in a **separate "Buy on" row below the existing 'Listen on' stream bar**
- All buy links open in a **new tab** — user stays on Mercury
- **Footer disclosure only** — no per-link affiliate badge or "sponsored" label. A site-wide affiliate disclosure in the footer is sufficient.

### Platform Coverage
- Five target platforms: **Bandcamp, Amazon Music, Apple Music, Beatport, Discogs Marketplace**
- Which affiliate programs to actually implement is **deferred to research** — researcher should investigate all five programs (availability, commission rates, signup requirements, URL structure) before final decision
- **All five platforms always shown** on release pages — no hiding a platform just because we lack a direct URL
- Platforms with a direct URL: show the direct product link (affiliate-coded)
- Platforms without a direct URL: show a pre-filled search fallback link (also affiliate-coded)

### Link Sourcing
- Buy link URLs are **constructed by Mercury** — not read from MusicBrainz purchase relationships as primary source. We detect the platform from artist/release metadata and construct appropriate URLs.
- Affiliate IDs stored in **environment variables / config file** — not hardcoded in source. Easy to update without redeployment.
- URL detection/construction happens **live on release page load** — same pattern as existing stream links. No pipeline changes required.
- Bandcamp's specific affiliate URL structure and program mechanics: **defer to researcher** — exact parameter format unknown.

### Missing Link Behavior (Search Fallbacks)
- Search fallback URL format: **pre-filled search** — links to platform's search page with artist + release name pre-populated (e.g., `amazon.com/s?k=Artist+Album+Name`)
- Affiliate codes applied to **both direct links and search fallbacks** — everything gets coded
- **Subtle visual distinction** between direct links (exact product page) and search fallbacks: small search icon or '?' indicator on the button so users know they're going to search, not a product page

### Claude's Discretion
- Exact visual design of the "Buy on" row (button style, spacing, icon choices beyond search/direct distinction)
- Tracklist display format on release pages
- Credits/personnel display format
- Error states and loading skeletons for release pages
- How to handle releases with no MusicBrainz data (extremely sparse metadata)

</decisions>

<specifics>
## Specific Ideas

- The "Buy on" row should feel like the "Listen on" bar — consistent visual language, just a different action row below it
- Search fallback buttons need a small visual cue (search icon or '?') so users aren't deceived into thinking they're going to a specific product page

</specifics>
