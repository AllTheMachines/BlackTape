## Summary

Phase 06.1 has two distinct jobs: (1) build release detail pages from scratch, and (2) wire up affiliate-coded buy links on those pages. The technical implementation is well-understood and follows existing Mercury patterns closely. The uncertain part is the affiliate program landscape.

**Affiliate program reality check:** Of the five target platforms, only Amazon Music (via Amazon Associates, 5% commission) and Apple Music (via Apple Services Performance Partners, ~7% on subscriptions / 2.5% on music purchases) have mature, self-serve affiliate programs with documented URL parameter formats. Bandcamp has no public affiliate program (their 2023 Songtradr acquisition disrupted any momentum toward one). Beatport had a program but shut it down in 2008 — third-party aggregators (Loopmasters/Post Affiliate Pro) run something adjacent but it's not a direct Beatport program. Discogs has never launched a public affiliate program despite years of user requests.

**Practical recommendation:** Build the affiliate infrastructure for Amazon and Apple Music first — these have working programs. For Bandcamp, Beatport, and Discogs, implement search fallback links now (no affiliate coding needed — just pre-filled search URLs). Add placeholder affiliate ID config slots so they can be enabled if programs emerge. The "Buy on" row still shows all five platforms with search fallbacks where no affiliate program exists.

**Release pages** are the bulk of the work. The MusicBrainz API provides everything needed: tracklist via `/ws/2/release/{mbid}?inc=recordings+artist-credits`, personnel/credits via artist-rels, cover art via Cover Art Archive. The release page route structure follows SvelteKit's filesystem pattern: `src/routes/artist/[slug]/release/[mbid]/+page.svelte`. Data loading mirrors the existing artist page pattern (server-side + universal load for Tauri).

**Primary recommendation:** Build release page infrastructure first, then add affiliate link construction as a layer on top. Use `platform.env` (Cloudflare) / `.dev.vars` (local) for affiliate IDs — never hardcode.

## Affiliate Program Research

### Platform-by-Platform Assessment

| Platform | Program Status | Commission | Signup | Affiliate URL Format | Confidence |
|----------|---------------|------------|--------|---------------------|------------|
| Amazon Music | ACTIVE (Amazon Associates) | 5% on music, $1.50–$7 on Unlimited signups | Self-serve at affiliate-program.amazon.com | `amazon.com/s?k={query}&tag={id}` | MEDIUM |
| Apple Music | ACTIVE (Performance Partners via Partnerize) | 7% music purchases / 100% first subscription month | Application required; limited acceptance | `geo.music.apple.com/...?at={token}&app=music&ls=1` | HIGH |
| Bandcamp | NO ACTIVE PROGRAM | N/A | N/A | N/A — use plain search fallback | MEDIUM |

| Discogs Marketplace | NO ACTIVE PROGRAM | N/A | N/A | N/A — use plain search fallback | HIGH |

### Bandcamp
Bandcamp does not have a public affiliate program that pays commission. They previously had nothing resembling one, and the 2023 Songtradr acquisition eliminated any trajectory toward building one. Some affiliate aggregator sites list Bandcamp with a "5-10%" rate, but this is unverified and no official program documentation exists. Bandcamp's artist pages ARE on artist subdomains (e.g., `artist.bandcamp.com`) — which means we can detect Bandcamp URLs from MusicBrainz data and use direct links when available, but they will be non-coded (no affiliate commission).

**URL patterns:**
- Direct artist page (from MusicBrainz URL relations): `https://artistname.bandcamp.com/album/release-slug`
- Search fallback: `https://bandcamp.com/search?q={artist}+{release}` — confirmed working, no affiliate parameter exists

**Decision: Show Bandcamp links when MusicBrainz has them (direct link). Show search fallback otherwise. No commission available.**

### Amazon Music
Amazon Associates is a mature, working program. Music category earns **5% commission** on qualifying purchases. The program also has a Bounty Program for Amazon Music Unlimited signups ($1.50–$7 per signup).

**URL patterns:**
- Search fallback: `https://www.amazon.com/s?k={artist}+{release}&tag={associates-id}`
- The `search-alias=digital-music` parameter can be added to narrow results to digital music
- Product page (when ASIN known): `https://www.amazon.com/dp/{ASIN}/?tag={associates-id}`

**Since we don't store ASINs, we'll always use search fallback:** `https://www.amazon.com/s?k={artist}+{release}&tag={id}`

**Signup:** affiliate-program.amazon.com — free, self-serve, approves immediately for most applicants. Requires an active website.

### Apple Music
Apple Services Performance Partners (via Partnerize) is a gated program. **Commission: 7% on songs/albums, 100% commission on first month of Apple Music subscription.** Application required — Apple reviews applicants and limits enrollment to partners who can drive volume.

**URL patterns (VERIFIED from official FAQ):**
- Affiliate link with token: `https://music.apple.com/us/album/{album-name}/{album-id}?at={token}&ct={campaign}&ls=1&app=music`
- Search fallback: `https://music.apple.com/us/search?term={artist}+{release}&at={token}&app=music`
- The `at=` parameter is the Partner Token from the Partnerize dashboard
- `app=music` is required for Apple Music links
- `ct=` is optional campaign tracking (max 40 chars, no `?`, `!`, or `&`)

**Since we don't store Apple Music catalog IDs, we'll use search fallback.**

**Signup:** performance-partners.apple.com — free to apply but selective acceptance. May need to establish traffic first.

### Beatport
Beatport ended their affiliate program in **2008**. The Beatport Group (parent: Loopmasters) runs an affiliate program through Post Affiliate Pro, but this covers Loopmasters/Splice-type products, not Beatport music downloads. No active affiliate program for Beatport releases exists.

**URL patterns:**
- Search (confirmed from live URL testing): `https://www.beatport.com/search?q={artist}+{release}`
- Artist page: `https://www.beatport.com/artist/{slug}/{id}` — requires Beatport-specific IDs we don't have

**Decision: Search fallback only, no affiliate coding possible for Beatport.**

### Discogs Marketplace
Discogs has never launched a public affiliate program. Community forum discussions dating to 2013 show users requesting one, with no action. No affiliate network carries Discogs. Partnership inquiries exist on their About page but these are ad/sponsorship deals, not CPA affiliate programs.

**URL patterns:**
- Search: `https://www.discogs.com/search/?q={artist}+{release}&type=release`
- Marketplace search: `https://www.discogs.com/sell/list?q={artist}+{release}` (shows vinyl for sale)

**Decision: Search fallback to Discogs marketplace search. No affiliate coding possible.**

## Release Page Architecture

### Route Structure

SvelteKit filesystem routing supports nested dynamic segments. The natural pattern for Mercury:

```
src/routes/
  artist/
    [slug]/
      +page.svelte            ← existing artist page
      +page.server.ts         ← existing (SSR)
      +page.ts                ← existing (universal/Tauri)
      release/
        [mbid]/
          +page.svelte        ← NEW: release detail page
          +page.server.ts     ← NEW: SSR data loading
          +page.ts            ← NEW: universal load (Tauri)
```

URL shape: `/artist/aphex-twin/release/59211ea4-ffd2-4ad9-9a4e-941d3148024a`

The release MBID is already available in the `ReleaseGroup` type (`release.mbid`), so linking from `ReleaseCard` to the release page requires no new data.

**IMPORTANT:** The existing `ReleaseGroup.mbid` is the release-GROUP MBID (not a specific release MBID). This is fine for linking but has implications for the tracklist API call (see below).

### Making ReleaseCard Clickable

Currently `ReleaseCard.svelte` shows release info and inline play buttons. To add navigation, wrap the cover art + title in an `<a href="/artist/{slug}/release/{release.mbid}">` link. The artist slug needs to be passed down from the artist page.

**Option A:** Pass `artistSlug` prop to ReleaseCard — simple, clean.
**Option B:** Derive from URL using `$page.params` inside ReleaseCard — less prop drilling but couples component to routing.

**Recommendation: Option A** — explicit prop, follows Mercury's existing patterns.

### Release Page Data Loading

The release page server load function follows the exact pattern of the existing artist page:

```typescript
// +page.server.ts for release/[mbid]
export const load: PageServerLoad = async ({ params, platform, fetch }) => {
  const { slug, mbid } = params;
  // 1. Fetch release details from MusicBrainz (tracklist, credits)
  // 2. Construct affiliate buy links
  // 3. Return all data
};
```

The Tauri universal load (`+page.ts`) needs to handle direct MusicBrainz API calls, same as the artist page's `+page.ts`.

### MusicBrainz API for Release Detail

**Key insight:** `ReleaseGroup.mbid` is the release-GROUP MBID. To get a tracklist, we need a specific release within that group.

**Two-step approach:**
1. Browse releases in the release-group: `GET /ws/2/release?release-group={mbid}&inc=artist-credits+media+recordings&limit=1&fmt=json` — gets the "main" release (usually the first/original edition)
2. The response includes full tracklist via `media[].tracks[]`

**Alternative (simpler but less optimal):** Use the release-group MBID to fetch the first release in one call:
```
GET https://musicbrainz.org/ws/2/release?release-group={release_group_mbid}&inc=recordings+artist-credits+media&limit=1&fmt=json
```

This returns `releases[0]` with full track data. **Confirmed:** the `/ws/2/release` browse endpoint supports `inc=recordings` which returns each track with title, position, length, and recording MBID.

**For personnel/credits:** Add `artist-rels+recording-level-rels` to the inc parameters — returns instrument/role credits per recording.

**Rate limiting:** Existing Mercury pattern uses 1100ms rate limiting and Cloudflare Cache API with 24hr TTL. Apply the same to release endpoint calls.

**Full API call for a complete release page:**
```
GET https://musicbrainz.org/ws/2/release?release-group={mbid}&inc=recordings+artist-credits+media+artist-rels&limit=1&fmt=json
```

### Cover Art

Mercury already uses Cover Art Archive for release-group artwork:
```
https://coverartarchive.org/release-group/{mbid}/front-250
```
The release page can use a larger variant: `front-500` or `front-1200` for the hero image.

Available sizes (VERIFIED from CAA API docs): **250px, 500px, 1200px** (and the deprecated aliases `small`/`large` for 250/500).

## Affiliate Link Construction

### Architecture: `src/lib/affiliates/` module

New module following the `src/lib/embeds/` pattern:

```
src/lib/affiliates/
  types.ts        — BuyLink interface, platform types, config interface
  config.ts       — reads affiliate IDs from environment
  construct.ts    — URL construction logic per platform
  index.ts        — public API
```

### BuyLink Type

```typescript
// types.ts
export type BuyPlatform = 'bandcamp' | 'amazon' | 'apple-music' | 'beatport' | 'discogs';

export interface BuyLink {
  platform: BuyPlatform;
  label: string;
  url: string;
  isDirect: boolean;     // true = known product URL; false = search fallback
}

export interface AffiliateConfig {
  amazonTag: string | null;        // Amazon Associates tag (e.g., "mercury-20")
  appleToken: string | null;       // Apple at= partner token
  appleCampaign: string | null;    // Optional ct= campaign name
}
```

### Environment Variable Access

In Cloudflare Pages + SvelteKit, server-side env vars from the Pages dashboard are accessed via `$env/dynamic/private` (NOT `$env/static/private` — CF Pages vars are only available at runtime).

```typescript
// config.ts
import { env } from '$env/dynamic/private';

export function getAffiliateConfig(): AffiliateConfig {
  return {
    amazonTag: env.AFFILIATE_AMAZON_TAG || null,
    appleToken: env.AFFILIATE_APPLE_TOKEN || null,
    appleCampaign: env.AFFILIATE_APPLE_CAMPAIGN || null,
  };
}
```

**Local dev:** Variables defined in `.dev.vars` file (Wrangler's secret file, same directory as `wrangler.jsonc`):
```
AFFILIATE_AMAZON_TAG=mercury-20
AFFILIATE_APPLE_TOKEN=your-token-here
AFFILIATE_APPLE_CAMPAIGN=mercury
```

**Production:** Set as encrypted secrets in Cloudflare Pages dashboard under Settings > Environment Variables.

**For Tauri desktop:** No server-side env available. Use a config file approach or simply omit affiliate IDs — Tauri pages can construct non-coded links and still show the buy buttons as useful search links.

### URL Construction Logic

```typescript
// construct.ts
export function buildBuyLinks(
  artistName: string,
  releaseTitle: string,
  bandcampUrl: string | null,   // direct URL from MusicBrainz if available
  config: AffiliateConfig
): BuyLink[] {
  const query = encodeURIComponent(`${artistName} ${releaseTitle}`);

  return [
    buildBandcampLink(bandcampUrl, artistName, releaseTitle),
    buildAmazonLink(query, config.amazonTag),
    buildAppleMusicLink(query, config.appleToken, config.appleCampaign),
    buildBeatportLink(artistName, releaseTitle),
    buildDiscogsLink(query),
  ];
}

function buildBandcampLink(directUrl: string | null, artist: string, title: string): BuyLink {
  if (directUrl) {
    return { platform: 'bandcamp', label: 'Bandcamp', url: directUrl, isDirect: true };
  }
  const q = encodeURIComponent(`${artist} ${title}`);
  return {
    platform: 'bandcamp',
    label: 'Bandcamp',
    url: `https://bandcamp.com/search?q=${q}`,
    isDirect: false
  };
}

function buildAmazonLink(query: string, tag: string | null): BuyLink {
  const base = `https://www.amazon.com/s?k=${query}`;
  const url = tag ? `${base}&tag=${tag}` : base;
  return { platform: 'amazon', label: 'Amazon Music', url, isDirect: false };
}

function buildAppleMusicLink(query: string, token: string | null, campaign: string | null): BuyLink {
  let url = `https://music.apple.com/us/search?term=${query}&app=music&ls=1`;
  if (token) url += `&at=${token}`;
  if (campaign) url += `&ct=${encodeURIComponent(campaign)}`;
  return { platform: 'apple-music', label: 'Apple Music', url, isDirect: false };
}

function buildBeatportLink(artist: string, title: string): BuyLink {
  const q = encodeURIComponent(`${artist} ${title}`);
  return {
    platform: 'beatport',
    label: 'Beatport',
    url: `https://www.beatport.com/search?q=${q}`,
    isDirect: false
  };
}

function buildDiscogsLink(query: string): BuyLink {
  return {
    platform: 'discogs',
    label: 'Discogs',
    url: `https://www.discogs.com/sell/list?q=${query}&type=release`,
    isDirect: false
  };
}
```

### Detecting Bandcamp URL from MusicBrainz Data

The existing `ReleaseGroup` type has a `links: ReleaseLink[]` field which contains streaming platform links. However, Bandcamp links on releases are not always present in MusicBrainz data. The release page server load should check `release.links` for a bandcamp URL and pass it to the buy link constructor.

Additionally, the existing artist page loads `links.bandcamp[]` (artist-level Bandcamp URLs). These are ARTIST page links (e.g., `artist.bandcamp.com`) not release-level — not appropriate for per-release buy links. Only use Bandcamp URLs that are specifically on the release's MusicBrainz URL relationships.

```typescript
import { buildBuyLinks } from '$lib/affiliates/construct';

// In load():
const affiliateConfig = getAffiliateConfig();
const bandcampUrl = releaseLinks.find(l => l.platform === 'bandcamp')?.url ?? null;
const buyLinks = buildBuyLinks(artist.name, release.title, bandcampUrl, affiliateConfig);
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Affiliate URL construction | Custom URL builder from scratch | Simple string template functions in `construct.ts` | URL construction is straightforward for these platforms — no library needed |
| Rate limiting for MB API | Custom timer | Existing module-level `lastMbRequest` pattern | Already proven in the codebase |
| Cloudflare caching | Custom cache | Existing `platform.caches` Cache API pattern | Already proven, zero-cost |
| Env var access | `process.env` directly | `$env/dynamic/private` from SvelteKit | CF Pages runtime vars only accessible this way |
| Type definitions | Inline objects | Typed interfaces in `affiliates/types.ts` | Consistency with existing `embeds/types.ts` pattern |

**Key insight:** Don't use any affiliate link management npm packages (Skimlinks, Viglink, etc.) — they inject JavaScript trackers and violate Mercury's no-tracking principle. All affiliate coding is done manually via URL parameters.

## Common Pitfalls

### Pitfall 1: Release Group vs. Release MBID Confusion
**What goes wrong:** `ReleaseGroup.mbid` (already in the codebase) is the release-GROUP MBID, not a specific release MBID. The Cover Art Archive endpoint works with both, but the MusicBrainz tracklist API requires a specific release MBID.
**How to avoid:** Use the browse endpoint: `GET /ws/2/release?release-group={mbid}&limit=1&fmt=json&inc=recordings+media` — this gives you the first release in the group, from which you can get tracks. The release-group MBID in the URL is fine for routing (stable identifier).
**Warning signs:** Getting empty `media` or `tracks` arrays from the API.

### Pitfall 2: `$env/static/private` vs `$env/dynamic/private` on Cloudflare Pages
**What goes wrong:** Using `$env/static/private` (build-time) causes affiliate IDs to be empty at runtime on Cloudflare Pages because CF Pages environment variables are runtime-only.
**How to avoid:** Always use `import { env } from '$env/dynamic/private'` for Cloudflare Pages server-side secrets.
**Warning signs:** Affiliate IDs are empty strings in production, work fine in Wrangler local dev only if `.dev.vars` is configured.

### Pitfall 3: Hardcoding Affiliate IDs
**What goes wrong:** IDs end up in git, get rotated by affiliate programs, can't be updated without redeployment.
**How to avoid:** Locked decision — env vars only. Config object created fresh per request from `$env/dynamic/private`.
**Warning signs:** Finding literal affiliate tag strings in TypeScript files.

### Pitfall 4: MusicBrainz Rate Limit on Release Page
**What goes wrong:** Release page loads hit MusicBrainz twice (once for artist page, once for release detail page navigation). The existing 1100ms rate limiter is module-level and shared — rapid navigation could cause 429 errors.
**How to avoid:** Apply the same Cloudflare Cache API caching pattern (24hr TTL) to the release detail API calls. Cached hits don't count against rate limit.
**Warning signs:** Occasional 502 errors from the MB API proxy.

### Pitfall 5: Apple Music Search URL Locale
**What goes wrong:** `music.apple.com/us/search` hardcodes US locale — non-US users may see different results or get redirected.
**How to avoid:** Consider using `geo.music.apple.com` which auto-redirects by geography, or accept that we default to US store (acceptable for MVP).
**Warning signs:** Apple Music links not working for international users.

### Pitfall 6: Bandcamp URL Source
**What goes wrong:** Using the artist-level Bandcamp URL (e.g., `artist.bandcamp.com`) as a release-level buy link — sends users to the artist homepage, not a specific release.
**How to avoid:** Only use Bandcamp URLs from the release's MusicBrainz URL relationships (not from the artist's relationships). Fall back to Bandcamp search when no release-specific URL is available.
**Warning signs:** Clicking Bandcamp buy link goes to artist homepage instead of the album.

### Pitfall 7: Discogs Marketplace vs Database Search
**What goes wrong:** Using `discogs.com/search?q=` sends users to the DATABASE (for cataloguing), not the MARKETPLACE (for buying). They want to buy vinyl, not edit entries.
**How to avoid:** Use `discogs.com/sell/list?q=...` or `discogs.com/search/?q=...&type=release` — the sell listing URL shows items available for purchase.
**Warning signs:** Users land on Discogs but can't find anything to buy.

## Open Questions

1. **Amazon product search category narrowing**
   - What we know: `s?k={query}&tag={id}` works for general Amazon search
   - What's unclear: Whether adding `&i=digital-music` (digital music category) helps or hurts for obscure/niche releases (may exclude physical media)
   - Recommendation: Start without category filter. Users discover both digital and physical options.

2. **Apple Music acceptance likelihood**
   - What we know: Apple limits enrollment to partners with "volume and quality"
   - What's unclear: Whether Mercury's traffic level will meet Apple's threshold at launch
   - Recommendation: Apply to the program early. If rejected, Apple links still work as non-coded search fallbacks — no functional degradation.

3. **Bandcamp URL detection from MusicBrainz release-group data**
   - What we know: Release-group URL-rels sometimes include Bandcamp purchase links; the existing `/api/artist/{mbid}/releases` endpoint already extracts these via `detectPlatform()`
   - What's unclear: What percentage of releases with Bandcamp presence actually have Bandcamp URLs in their MB release-group relationships vs. only at the artist level
   - Recommendation: Use what's available from existing data structures. Research during implementation — if hit rate is low, the search fallback is adequate.

4. **Discogs "buy" vs "search" URL for best UX**
   - What we know: `discogs.com/sell/list?q=...` shows marketplace listings; `discogs.com/search?q=...` shows the database
   - What's unclear: Which is more useful for users wanting to buy physical releases
   - Recommendation: Use `discogs.com/sell/list?q={query}&type=release` — this is the buying experience, not the cataloguing experience.

5. **MusicBrainz personnel credits depth**
   - What we know: `artist-rels` on a release gives production credits; `recording-level-rels` gives track-level credits
   - What's unclear: How rich the credits data actually is for typical releases in MusicBrainz
   - Recommendation: Fetch `artist-rels` as part of the release call. Display what's there; don't error on empty credits. Phase 7 (Knowledge Base / Liner Notes) will go deeper on credits.
