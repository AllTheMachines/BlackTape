## Summary

Phase 10.1 is a precision gap-closure phase with four well-scoped, independently fixable problems. The v1.0 audit identified the exact files, variables, and functions involved — this phase is implementation work, not design work. Three of the four fixes are frontend-only; one requires a new Rust command.

**GAP-05** (CSS variables) is a one-file, four-line addition to `src/lib/styles/theme.css` — adding alias variables that Phase 9/10 components assumed existed. The audit has already identified the exact 4 variables and their correct alias targets. Secondary note: `--spacing-xs/sm/md/lg/xl` (used in 4 Phase 9 files) also need aliases to `--space-*` but were not counted in the audit's "67 usages" because the rendering consequence (zero spacing vs invisible color) is subtler.

**GAP-06** (DM conversation list) requires one new `ConversationList.svelte` component plus a modification to `ChatOverlay.svelte` to render it in the 'dms' view. The entire data layer (`dmState.conversations`, `sendDM`, `markConversationRead`) is already implemented in `dms.svelte.ts`. The chatState already has `view = 'dm-thread'` and `activeConversationPubkey` fields waiting to be set. This is pure UI work — no store or backend changes needed.

**GAP-07** (kind:30078 taste publishing) requires a new `publishTasteProfile()` function in the comms layer and a call site + status indicator in `profile/+page.svelte`. The NDK publish pattern is established by `sendDM()` and `createRoom()`. The event content shape is already defined in `ai-taste-bridge.ts` — the consumer side already parses `{ topTags, topArtists }`.

**GAP-08** (export_play_history_to_path Rust command) requires adding one new function to `taste_db.rs` and registering it in `lib.rs generate_handler!`. The `export_play_history` command already fetches and serializes the data — the new command just needs to accept a path and write the JSON. The frontend call already exists in `history.ts:74`.

**Primary recommendation:** Implement in sequence — GAP-05 first (unblocks visual testing of all Phase 9/10 UI), GAP-06 second (closes COMM-04), GAP-07 third (enhances COMM-04 quality), GAP-08 fourth (closes export gap).

### Pattern 3: Taste Profile Publishing (kind:30078)

**What:** A new `publishTasteProfile()` function that publishes the user's taste data as a Nostr replaceable event (kind:30078, NIP-78 "arbitrary custom app data"), then a call site on the Profile page with a subtle status indicator.

**Location:** New function in `src/lib/comms/nostr.svelte.ts` or a new file `src/lib/comms/taste-publish.ts`. Recommend a new file to keep `nostr.svelte.ts` focused on init/connection.

**Event shape (already defined by consumer `ai-taste-bridge.ts:81-104`):**
```typescript
// kind:30078 with 'd' tag 'mercury-taste-profile'
// Content is JSON:
{
  topTags: string[],      // top 8 taste tags by weight
  topArtists: string[]    // top 5 favorite artist names
}
```

**NDK publish pattern (from existing `createRoom` in rooms.svelte.ts and `sendDM` in dms.svelte.ts):**
```typescript
import { ndkState } from './nostr.svelte.js';

export async function publishTasteProfile(): Promise<void> {
  const { ndk } = ndkState;
  if (!ndk || !ndkState.connected) return; // silent skip per user decision

  const { tasteProfile } = await import('$lib/taste/profile.svelte.js');
  if (!tasteProfile.isLoaded || tasteProfile.tags.length === 0) return;

  const { NDKEvent } = await import('@nostr-dev-kit/ndk');

  const topTags = [...tasteProfile.tags]
    .sort((a, b) => b.weight - a.weight)
    .slice(0, 8)
    .map(t => t.tag);

  const topArtists = tasteProfile.favorites
    .slice(0, 5)
    .map(f => f.artist_name);

  const event = new NDKEvent(ndk);
  event.kind = 30078;
  event.content = JSON.stringify({ topTags, topArtists });
  event.tags = [['d', 'mercury-taste-profile']];

  await event.publish();
}
```

**Trigger (Claude's discretion — user approved "auto-publish on Profile page load if Nostr is connected"):**
Call `publishTasteProfile()` in `profile/+page.svelte` `onMount`, after `initNostr()` completes or inside the existing Tauri `onMount` block, fire-and-forget.

**Status indicator (user decision: "subtle, near the Nostr key display section"):**
Profile page currently has no Nostr key display section — this is a new addition. The indicator should show user's npub (truncated) + publish status. State pattern: `let nostrStatus = $state<'idle' | 'publishing' | 'published' | 'no-nostr'>('idle')`. Wrap `publishTasteProfile()` with status updates.

**Silent skip rule (user decision):** If `!ndkState.connected`, set `nostrStatus = 'no-nostr'` and don't show indicator at all (or show nothing).

### Pattern 4: export_play_history_to_path Rust Command

**What:** A new Tauri command in `taste_db.rs` that accepts `path: String` and writes the JSON from `export_play_history` to disk, returning `{ path, count }`.

**Exact call site in frontend (confirmed at `history.ts:74`):**
```typescript
await invoke('export_play_history_to_path', { path, json });
```
The frontend already passes `{ path, json }` — the Rust command must accept both.

**Recommended Rust implementation:**
```rust
#[derive(Debug, Serialize)]
pub struct ExportResult {
    pub path: String,
    pub count: usize,
}

#[tauri::command]
pub fn export_play_history_to_path(
    path: String,
    json: String,
    state: tauri::State<'_, TasteDbState>,
) -> Result<ExportResult, String> {
    // Count records for useful return value
    let count: usize = serde_json::from_str::<Vec<serde_json::Value>>(&json)
        .map(|v| v.len())
        .unwrap_or(0);

    // Overwrite behavior (Claude's discretion — pragmatic default)
    std::fs::write(&path, json.as_bytes())
        .map_err(|e| format!("Failed to write history to '{}': {}", path, e))?;

    Ok(ExportResult { path, count })
}
```

**Note on state parameter:** The frontend call passes `{ path, json }` — no state access needed. But adding `state` parameter is harmless if needed; actually it is NOT needed since we're just doing a file write. Omit `state` if not querying taste.db. Compare with `write_json_to_path` which has no state parameter.

**Registration in lib.rs:** Add `ai::taste_db::export_play_history_to_path` to the `generate_handler!` macro list after `ai::taste_db::write_json_to_path`.

**Existing analog (`write_json_to_path`):** Already does the same write but without returning count. The new command is a specialized version with a richer return type and a count-before-write step.

**Frontend integration — Profile page export button (user decision):** Add an "Export Play History" button near the listening history section. The button should call `exportPlayHistory()` from `$lib/taste/history` (which already handles the full save dialog + invoke flow). The export button on Profile is separate from the one in Settings > Listening History — both call the same underlying function.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| npub encoding | Custom bech32 encoder | `nip19.npubEncode()` from nostr-tools | NDK already ships nostr-tools as a dependency; bech32 encoding has edge cases |
| npub decoding | Custom bech32 decoder | `nip19.decode()` from nostr-tools | Same as above; handles checksum validation |
| Pubkey hex validation | Custom regex | `/^[0-9a-f]{64}$/i.test(value)` | Simple regex is correct here; 64 hex chars is the full validation needed |
| JSON file writing in Rust | Custom file path logic | `std::fs::write` | Already used by `save_base64_to_file` and `write_json_to_path` — established pattern |

## Common Pitfalls

### Pitfall 1: --spacing-* vs --space-* — Missing from the 4-alias fix
**What goes wrong:** Adding only the 4 audit-identified aliases (`--bg-primary`, `--bg-tertiary`, `--border`, `--accent`) leaves `--spacing-xs/sm/md/lg/xl` still undefined in 4 Phase 9 files. These will resolve to `0` (zero spacing) making profile sections collapse.
**Why it happens:** The audit counted 67 broken color/border usages but `--spacing-*` usages are fewer and their visual consequence (zero gap vs invisible color) was noted separately.
**How to avoid:** Add 5 additional spacing aliases in the same CSS block.
**Warning signs:** Profile page sections appear with no gap between them even after the 4-alias fix.

### Pitfall 2: ConversationList importing from notifications.svelte.ts creates circular dep
**What goes wrong:** ChatPanel and ChatOverlay both import from `notifications.svelte.ts`. If ConversationList also imports from `dms.svelte.ts` AND `notifications.svelte.ts`, this can trigger circular module resolution.
**Why it happens:** The comms modules were deliberately structured to prevent this — `notifications.svelte.ts` has a comment explaining it deliberately does NOT import dms/rooms.
**How to avoid:** ConversationList should import `chatState` from `notifications.svelte.ts` and `dmState`, `markConversationRead` from `dms.svelte.ts`. This is safe — it's the same pattern ChatPanel already uses.
**Warning signs:** TypeScript error about circular dependency or `chatState` being undefined at runtime.

### Pitfall 3: ChatOverlay 'dms' view split breaks 'dm-thread' navigation
**What goes wrong:** If the ChatOverlay restructure is not careful, the back-navigation from 'dm-thread' to 'dms' (conversation list) may not work because the 'dms' branch now renders a different component.
**Why it happens:** The current `{:else}` catch-all handled both. After the split, a "Back" button in ChatPanel for dm-thread needs to set `chatState.view = 'dms'`.
**How to avoid:** Add a back button to ChatPanel when `chatState.view === 'dm-thread'` — `onclick={() => chatState.view = 'dms'}`. Check ChatPanel.svelte for where to add this.
**Warning signs:** Once in a DM thread, user cannot get back to conversation list without closing overlay.

### Pitfall 4: kind:30078 published with wrong 'd' tag breaks peer fetching
**What goes wrong:** The consumer in `ai-taste-bridge.ts` fetches with `'#d': ['mercury-taste-profile']`. If the publisher uses a different 'd' tag value (e.g., 'mercury-taste'), the event is never found.
**Why it happens:** The d-tag value must match exactly between publisher and subscriber.
**How to avoid:** Use exactly `['d', 'mercury-taste-profile']` as confirmed at `ai-taste-bridge.ts:91`.
**Warning signs:** AI taste bridge always falls back to "Peer hasn't published a taste profile" even for the local user testing with their own pubkey.

### Pitfall 5: export_play_history_to_path state parameter mismatch
**What goes wrong:** Adding `state: tauri::State<'_, TasteDbState>` to the Rust function signature when the frontend call only passes `{ path, json }` causes a Tauri invoke error.
**Why it happens:** Tauri command parameter injection from managed state is automatic, but only if the state parameter name matches Tauri's injection conventions. However the real issue is that if state is declared but the function doesn't need it, it's harmless — Tauri still injects it. But double-locking the `TasteDbState` Mutex from two commands at the same time would panic.
**How to avoid:** Do NOT lock `TasteDbState` in `export_play_history_to_path` — the function only writes a file, not the database. Omit the `state` parameter entirely, same as `write_json_to_path` and `save_base64_to_file`.

### Pitfall 6: nostr-tools nip19 may not be directly importable
**What goes wrong:** `import { nip19 } from 'nostr-tools'` may fail if nostr-tools is not in the direct dependency tree (only as a transitive NDK dep).
**Why it happens:** Vite/Rollup only bundle explicitly declared dependencies for ESM; transitive deps may not be importable by name.
**How to avoid:** Check `package.json` for nostr-tools. If absent, use a simpler truncation approach: show first 8 hex chars + "..." + last 4 hex chars. This is readable without the npub encoding step.

### ConversationList.svelte — Core Structure (GAP-06)

```svelte
<script lang="ts">
  import { dmState, markConversationRead } from '$lib/comms/dms.svelte.js';
  import { chatState } from '$lib/comms/notifications.svelte.js';

  let newPubkeyInput = $state('');
  let pubkeyError = $state('');

  function openConversation(peerPubkey: string) {
    markConversationRead(peerPubkey);
    chatState.activeConversationPubkey = peerPubkey;
    chatState.view = 'dm-thread';
  }

  function truncatePubkey(hex: string): string {
    // Try npub encoding if nostr-tools available, else hex truncation
    return hex.slice(0, 8) + '...' + hex.slice(-4);
  }

  function getLastMessage(conv: DmConversation): string {
    const last = conv.messages[conv.messages.length - 1];
    if (!last) return '';
    return last.content.length > 50 ? last.content.slice(0, 50) + '…' : last.content;
  }

  function handleNewDm() {
    const val = newPubkeyInput.trim();
    let hex = '';
    if (val.startsWith('npub1')) {
      // decode npub — use nip19 or show error
      try {
        const { nip19 } = await import('nostr-tools'); // dynamic — may fail
        const decoded = nip19.decode(val);
        if (decoded.type !== 'npub') throw new Error('not npub');
        hex = decoded.data as string;
      } catch {
        pubkeyError = 'Invalid pubkey format';
        return;
      }
    } else if (/^[0-9a-f]{64}$/i.test(val)) {
      hex = val.toLowerCase();
    } else {
      pubkeyError = 'Invalid pubkey format';
      return;
    }
    pubkeyError = '';
    newPubkeyInput = '';
    chatState.activeConversationPubkey = hex;
    chatState.view = 'dm-thread';
  }
</script>
```

**Note on async `handleNewDm`:** The nostr-tools import must be dynamic inside async. Since this uses nip19.decode which is sync after import, this is fine.
