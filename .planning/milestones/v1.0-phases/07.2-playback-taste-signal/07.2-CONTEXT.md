# Phase 07.2: Playback → Taste Signal - Context

**Gathered:** 2026-02-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Local audio playback (and embedded player events where feasible) automatically feeds the taste profile. When you listen to tracks, that listening history becomes taste input — no manual favoriting required. This closes the gap where taste only came from explicit signals (favorites, settings). Closes GAP-03 from v1.0 audit, satisfies PLAYER-03.

</domain>

<decisions>
## Implementation Decisions

### Signal counting threshold
- A play counts when the user has listened to **70-75% of the track**
- Anything less than that is treated as a skip — not counted

### Repeat plays
- Each qualifying listen of the same track **adds weight** — no ceiling
- Obsessing over a track is a meaningful signal; repeated listens compound

### Activation threshold
- Taste profile is not affected until **5 qualifying plays** have accumulated
- Below that threshold, plays are recorded but don't yet influence recommendations
- Activation is **silent** — no notification or toast when the threshold is crossed

### Play decay
- **Recent plays weigh more than older ones** — time-based decay applied
- Old listening history still counts, but recent sessions carry more influence

### Embed player coverage
- Attempt to hook **embedded player events** in addition to local file playback
- SoundCloud widget API and YouTube iframe API are candidates — researcher to assess feasibility per platform
- Local file playback (Tauri audio engine) is the **guaranteed baseline**; embeds are best-effort

### Genre/tag impact
- Plays shape **both artist AND genre/tag taste** — not just the artist side
- Tags associated with listened tracks feed into genre weighting (e.g. listening to post-punk tracks nudges genre weighting toward post-punk)

### Relative weighting vs explicit signals
- **Claude's discretion:** relative weight ratio of plays vs manual favorites (choose what makes the taste profile feel accurate)
- **Claude's discretion:** whether plays stack additively on top of an existing explicit favorite or not

### User visibility
- Listening history is **visible on the taste profile page**
- The taste profile page shows **both**: a track history list AND how listening shifted taste (boosted artists/tags)
- Claude determines whether to add a new section to an existing taste profile page, or create one — check what currently exists

### History storage
- **Local only** — listening history never leaves the device, no cloud sync

### User controls
- **Delete individual plays** from history — users can remove specific listens
- **Clear all history** — nuclear option to wipe everything and start fresh
- **Private listening mode** — toggle to listen without plays counting toward taste (incognito)
- **Export** — listening history is exportable (CSV or JSON) — user's data is theirs

### Claude's Discretion
- Exact weight ratio of plays vs manual favorites
- Whether/how plays stack on top of an existing explicit favorite
- Decay curve design (half-life, formula, etc.)
- Taste recompute cadence (per track vs batched)
- Temp file handling and storage schema for play history

</decisions>

<specifics>
## Specific Ideas

- "Each listen adds weight" — repeated listening is a strong positive signal, not noise
- Private listening mode (incognito) was explicitly requested — important for the privacy model
- Both individual delete AND full clear are needed — not just one option
- Listening history export is required — "your data is yours" principle

</specifics>
