  <action>
    **src/routes/explore/+page.ts:** Minimal load function. Returns empty object. All logic is client-side.

    **src/routes/explore/+page.svelte:** The NL explore experience:

    **Layout structure:**
    - Page title area: "Explore" heading with subtle subtext — user's top 3 taste genres shown as context (e.g., "Your taste leans toward ambient, electronic, experimental" — only if taste data exists). This is the "subtle hints" per user decision.
    - Main query input: Large, prominent text input. Placeholder: "Find me something like Boards of Canada but darker..." Monospace-esque font for the input. Submit on Enter.
    - Results area: Scrollable list of ExploreResult components.
    - Refinement input: Appears below results after first query. Smaller input. Placeholder: "Refine — try 'darker', 'more electronic', 'from the 90s'..." Submit on Enter.
    - Conversation counter: Small muted text "3 of 5 refinements" when in refinement mode. After 5 refinements, show "Start a new search" prompt and disable refinement input.

    **State management:**
    ```typescript
    let query = $state('');
    let results = $state<Array<{ number: number; artistName: string; description: string; slug: string | null }>>([]);
    let isLoading = $state(false);
    let refinementInput = $state('');
    let conversationHistory = $state<Array<{ role: string; content: string }>>([]);
    let refinementCount = $state(0);
    const MAX_REFINEMENTS = 5;
    ```

    **Query flow:**
    1. User types query, presses Enter.
    2. Set isLoading = true. Show standard loading skeleton (3-4 skeleton lines, same animation as other loading states).
    3. Get AI provider. If not ready, show message "AI features are not enabled. Enable them in Settings."
    4. Build prompt: if tasteProfile has data, use `nlExploreWithTaste` variant. Otherwise use `nlExplore`.
    5. Call provider.complete() with prompt. Temperature 0.8, maxTokens 1024.
    6. Parse response: Split by numbered lines (regex: /^\d+\.\s*\*?\*?(.+?)\*?\*?\s*[—\-–]\s*(.+)/gm). Extract artist name + description.
    7. For each parsed artist: Try to find in mercury.db via search (dynamic import TauriProvider, quick exact search). If found, attach slug. If not, slug = null.
    8. Populate results array with parsed data.
    9. Store the query and response in conversationHistory.
    10. Show results, show refinement input.

    **Refinement flow:**
    1. User types refinement, presses Enter.
    2. Build refined prompt using `nlRefine` with: original query, summary of previous results, refinement text.
    3. Call provider.complete(). Parse same way.
    4. Increment refinementCount. Replace results with new ones.
    5. Add to conversationHistory.
    6. If refinementCount >= MAX_REFINEMENTS, disable refinement input, show "Start a new search" button.

    **"Start new search" button:** Clears results, query, refinement, conversationHistory. Focuses main input.

    **Non-Tauri fallback:** Show centered message "Natural-language exploration is available in the Mercury desktop app." with link to download (or just the message).

    **AI not ready fallback:** Show centered message "Enable AI features in Settings to explore music with natural language." with link to /settings.

    **Styling:**
    - Match Mercury dark theme throughout.
    - Main query input: Large (1.2rem font), full-width, bg-elevated, border-subtle. Prominent but clean.
    - Results list: Left-aligned, generous spacing between items.
    - Refinement input: Smaller (0.95rem), full-width, appears with a fade-in after results load.
    - Page max-width matches other pages (var(--max-width)).
    - Loading skeleton matches existing patterns: 3-4 animated lines with subtle pulse.
  </action>
