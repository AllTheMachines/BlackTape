### Locked Decisions

**Signal counting threshold**
- A play counts when the user has listened to 70-75% of the track
- Anything less than that is treated as a skip — not counted

**Repeat plays**
- Each qualifying listen of the same track adds weight — no ceiling
- Obsessing over a track is a meaningful signal; repeated listens compound

**Activation threshold**
- Taste profile is not affected until 5 qualifying plays have accumulated
- Below that threshold, plays are recorded but don't yet influence recommendations

**Play decay**
- Recent plays weigh more than older ones — time-based decay applied
- Old listening history still counts, but recent sessions carry more influence

## Summary

This phase closes GAP-03: playback does not currently feed the taste profile. The audio engine (`audio.svelte.ts`) has a `timeupdate` event listener that already exposes `currentTime` and `duration` through `playerState`, but there is no hook that fires when a track reaches the 70-75% completion threshold. The taste signal computation (`signals.ts`) already has `recomputeTaste()` that persists tags to `taste.db`, but it only reads from library tracks and favorites — it has no concept of individual play events.

The primary work is: (1) add a `play_history` table to `taste.db` via a new Rust schema migration, (2) add corresponding Tauri commands for CRUD + export, (3) intercept the `timeupdate` event in `audio.svelte.ts` to detect the 70-75% threshold and record a qualifying play, (4) extend `recomputeTaste()` in `signals.ts` to weight play-history data with time-decay, and (5) surface the history and controls on the taste profile page (currently embedded in Settings). Private listening mode is a global toggle stored in `taste.db` (like other AI settings) and checked before recording any play.

Embed player events (SoundCloud, YouTube) are feasible but carry platform-specific caveats. SoundCloud has an official Widget JS API using `postMessage`; its `PLAY_PROGRESS` event gives `relativePosition` (0–1) making 0.70–0.75 detection straightforward. YouTube iframe API works via `YT.PlayerState.ENDED` but has a known Tauri-specific issue (Error 153 in production due to missing HTTP Referer header). Both are best-effort: the local audio engine is the only guaranteed signal path.

**Primary recommendation:** Hook `audio.svelte.ts` `timeupdate` for local plays (guaranteed baseline), add a new `playback` source to `recomputeTaste()` with time-decay weighting, and attempt SoundCloud widget binding as secondary signal. Defer YouTube embed signal until the Tauri referer header issue is resolved upstream.

### Pattern 1: Threshold Detection in audio.svelte.ts

**What:** Intercept the existing `timeupdate` handler to detect when `currentTime / duration >= 0.70`. Fire once per track load. Reset on `src` change.

**When to use:** Every local file play. This is the guaranteed signal path.

**Example:**

```typescript
// In audio.svelte.ts — additions to initAudio()

let thresholdFired = false;  // module-level, reset on new track

audio.addEventListener('timeupdate', () => {
    playerState.currentTime = audio!.currentTime;

    // Threshold detection: 70% of track duration
    const dur = audio!.duration;
    if (!thresholdFired && dur > 0 && audio!.currentTime / dur >= 0.70) {
        thresholdFired = true;
        // Fire-and-forget: do not await, do not block timeupdate
        import('./playback.svelte').then(({ recordQualifyingPlay }) => {
            recordQualifyingPlay(playerState.currentTrack!);
        });
    }
});

// Reset on new track load
audio.addEventListener('loadedmetadata', () => {
    playerState.duration = audio!.duration;
    playerState.isLoading = false;
    thresholdFired = false;  // reset for new track
});
```

### Pattern 3: Time-Decay Weight in Taste Computation

**What:** When computing taste from play history, recent plays get full weight, older plays get exponentially less weight. Claude's discretion: use a half-life of 30 days.

**Formula (recommended):**

```typescript
// In signals.ts — computeTasteFromPlayHistory()

const HALF_LIFE_DAYS = 30;
const HALF_LIFE_MS = HALF_LIFE_DAYS * 24 * 60 * 60 * 1000;

function decayWeight(playedAtMs: number): number {
    const ageMs = Date.now() - playedAtMs;
    // Exponential decay: weight = e^(-0.693 * age / halfLife)
    // 0.693 ≈ ln(2) — gives exactly 0.5 weight at half-life
    return Math.exp((-0.693 * ageMs) / HALF_LIFE_MS);
}

export async function computeTasteFromPlayHistory(): Promise<Map<string, number>> {
    const tagWeights = new Map<string, number>();
    const invoke = await getInvoke();

    // Fetch all history (no limit — decay handles old plays gracefully)
    const history = await invoke<PlayRecord[]>('get_play_history', { limit: null });

    // Group by artist to deduplicate tag lookups
    const artistAccum = new Map<string, number>();
    for (const play of history) {
        if (!play.artist_name) continue;
        const weight = decayWeight(play.played_at * 1000);
        artistAccum.set(
            play.artist_name,
            (artistAccum.get(play.artist_name) || 0) + weight
        );
    }

    // Look up tags for each unique artist and accumulate
    for (const [artistName, totalWeight] of artistAccum) {
        const tags = await lookupArtistTags(artistName);
        for (const tag of tags) {
            tagWeights.set(tag, (tagWeights.get(tag) || 0) + totalWeight);
        }
    }

    return tagWeights;
}
```

**Weight ratio (Claude's discretion):** Playback source is weighted at 1.0x (same as library). Manual favorites remain 2x. Rationale: a qualifying play (70%+) signals real engagement, equivalent to having the artist in your library, but one listen is less definitive than explicitly hearting them.

**Stacking on existing favorites (Claude's discretion):** Play weights are additive on top of favorites, not replacing. The merged weights sum before normalization. An artist both favorited AND frequently listened to gets compounding weight — correct behavior.

### Pattern 5: Private Listening Mode

**What:** A boolean toggle stored in `ai_settings` (key: `private_listening`). When `true`, `recordQualifyingPlay()` is a no-op. Reactive state in a new `playback.svelte.ts`.

**Why ai_settings:** Already has key-value storage in taste.db with `get_ai_setting`/`set_ai_setting`. No new Rust table needed.

**Example:**

```typescript
// src/lib/player/playback.svelte.ts (NEW FILE)
// Uses .svelte.ts extension for Svelte 5 $state runes

export const playbackState = $state({
    privateMode: false,
    totalQualifyingPlays: 0
});

export async function loadPlaybackSettings(): Promise<void> {
    const { invoke } = await import('@tauri-apps/api/core');
    const value = await invoke<string | null>('get_ai_setting', { key: 'private_listening' });
    playbackState.privateMode = value === 'true';
    const count = await invoke<number>('get_play_count');
    playbackState.totalQualifyingPlays = count;
}

export async function togglePrivateMode(): Promise<void> {
    const { invoke } = await import('@tauri-apps/api/core');
    const newVal = !playbackState.privateMode;
    await invoke('set_ai_setting', { key: 'private_listening', value: String(newVal) });
    playbackState.privateMode = newVal;
}

export async function recordQualifyingPlay(track: PlayerTrack): Promise<void> {
    if (playbackState.privateMode) return;  // incognito — do nothing

    const { invoke } = await import('@tauri-apps/api/core');
    await invoke('record_play', {
        trackPath: track.path,
        artistName: track.artist,
        trackTitle: track.title,
        albumName: track.album,
        durationSecs: track.durationSecs
    });

    playbackState.totalQualifyingPlays++;

    // Trigger taste recompute after each qualifying play
    // (per-track cadence is fine for personal library sizes — no perf concern)
    const { recomputeTaste } = await import('$lib/taste/signals');
    await recomputeTaste();

    // Refresh taste status (enables recommendations if now >= 5 plays)
    const { refreshTasteStatus } = await import('$lib/taste/profile.svelte');
    await refreshTasteStatus();
}
```

### Pattern 6: SoundCloud Widget Event Hook (best-effort embed signal)

**What:** After the SoundCloud oEmbed HTML is injected into `EmbedPlayer.svelte`, detect the iframe and attach SC Widget API listeners.

**Feasibility:** HIGH for SoundCloud specifically. The Widget API uses `window.postMessage` which works in Tauri's webview (CSP is null for Tauri per existing project decision). The `PLAY_PROGRESS` event includes `relativePosition` (0–1 range) — exactly what we need for 70% threshold detection.

**Loading pattern:** Load the SC Widget script once via `<script src="https://w.soundcloud.com/player/api.js">` (can be done conditionally when SoundCloud embed is revealed). This is a runtime dynamic load, no build change needed.

**Example (inside EmbedPlayer.svelte):**

```typescript
// After soundcloudEmbedHtml is rendered and iframe exists in DOM:
async function hookSoundCloudWidget() {
    // Ensure SC Widget API is loaded
    if (!window.SC) {
        await loadScript('https://w.soundcloud.com/player/api.js');
    }

    const iframe = el.querySelector('iframe');  // the rendered SC iframe
    if (!iframe) return;

    const widget = window.SC.Widget(iframe);
    let progressFired = false;

    widget.bind(window.SC.Widget.Events.PLAY, () => {
        progressFired = false;  // reset on new play
    });

    widget.bind(window.SC.Widget.Events.PLAY_PROGRESS, (data: { relativePosition: number }) => {
        if (!progressFired && data.relativePosition >= 0.70) {
            progressFired = true;
            // Note: SC embed plays artist + track but we only have the embed URL context
            // Record as best-effort with what we know
            import('$lib/player/playback.svelte').then(({ recordEmbedPlay }) => {
                recordEmbedPlay({ artistName: /* from embed context */ });
            });
        }
    });
}
```

**Caveat:** SoundCloud embed context in `EmbedPlayer.svelte` doesn't pass artist name directly — it's rendered from raw oEmbed HTML. The artist name must be passed as a prop from the parent (artist page) or extracted from the embed URL. This is a planner decision: pass artist info as a prop to EmbedPlayer.

### Pattern 7: YouTube Embed Signal (DEFER)

**Why defer:** Tauri production builds exhibit Error 153 ("Video player configuration error") from YouTube's iframe API due to the `tauri://localhost` protocol lacking a valid HTTP Referer header. This is a known upstream Tauri issue (github.com/tauri-apps/tauri/issues/14422). The YouTube iframe API's `onStateChange` / `YT.PlayerState.ENDED` would work conceptually but the YouTube player itself fails to load reliably in Tauri production. Do not implement YouTube embed signal in this phase.

### Anti-Patterns to Avoid

- **Blocking timeupdate with async work:** `timeupdate` fires ~4x/second. Never await inside it. Always fire-and-forget with `.then()` for any async recording work.
- **Storing play history in library.db:** Keep taste-related data in `taste.db` (project pattern established in STATE.md). library.db is for scanner/track metadata only.
- **Re-running full taste recompute on every timeupdate tick:** Recompute should fire once per qualifying play completion, not continuously. The `thresholdFired` flag prevents repeated triggers.
- **Exposing play history in web build:** This is Tauri-only. All history-related invokes must be inside dynamic imports with isTauri() or try/catch pattern.
- **Adding embed event hooks before the iframe is revealed:** EmbedPlayer uses click-to-load pattern. Bind widget events only after the iframe renders (after `revealEmbed(key)` call).

## Common Pitfalls

### Pitfall 1: Seek-Reset Race Condition

**What goes wrong:** User plays 80% of track, seeks back to 10%, then exits. `thresholdFired` is already `true` — no double-count. But if user seeks FORWARD past 70% on a new play of the same track (repeat mode), the flag is still true from the prior listen.

**Why it happens:** `thresholdFired` is reset on `loadedmetadata` (new src), but NOT on `repeat-one` mode replay (same src, just `currentTime` reset to 0).

**How to avoid:** Also reset `thresholdFired` on the `play` event IF `currentTime < 1` (indicating a restart from beginning). This catches repeat-one replays without losing the flag mid-track.

**Warning signs:** Play count not incrementing on `repeat-one` mode even after qualifying listens.

### Pitfall 2: Duration = 0 / NaN at First timeupdate

**What goes wrong:** `loadedmetadata` hasn't fired yet, `audio.duration` is NaN or 0. Division produces NaN >= 0.70 = false — safe. But duration could also be Infinity for live streams.

**Why it happens:** Short audio latency between `play()` and metadata load.

**How to avoid:** Guard with `!isNaN(dur) && isFinite(dur) && dur > 0` before the threshold check. This is already implicitly handled if duration defaults to 0 in playerState.

### Pitfall 3: Taste Recompute Cost with Large History

**What goes wrong:** After 1000 plays, `computeTasteFromPlayHistory()` issues 1000 artist lookups sequentially (matching how `computeTasteFromLibrary` works per-artist).

**Why it happens:** Current `lookupArtistTags` is one SQL query per artist. With large play history this compounds.

**How to avoid:** Group plays by unique artist name BEFORE looking up tags (as shown in Pattern 3 code). With grouping, you only need one DB query per unique artist — regardless of how many times that artist appears in history.

### Pitfall 4: Private Mode Not Loaded Before First Play

**What goes wrong:** User starts playing before `loadPlaybackSettings()` has resolved. Private mode defaults to `false` — play gets recorded even if user had previously toggled private mode on.

**Why it happens:** Async load of settings from taste.db takes a moment.

**How to avoid:** Load `playbackSettings` (including private mode) in the root layout's `onMount` alongside `loadTasteProfile()`. Both are fire-and-forget Tauri invokes — add `loadPlaybackSettings()` call there.

### Pitfall 5: SoundCloud Widget Script Double-Load

**What goes wrong:** Multiple artist pages with SoundCloud embeds — each tries to load `api.js`. Results in multiple SC widget instances or script-load errors.

**Why it happens:** No guard on dynamic script injection.

**How to avoid:** Check `if (!window.SC)` before injecting the script tag. The SC Widget API is a singleton on `window.SC` — safe to reuse once loaded.

### Pitfall 6: History Page vs Settings Page

**What goes wrong:** Building a whole new `/history` or `/taste-profile` route when the Settings page already hosts the Taste Editor section.

**Why it happens:** CONTEXT.md says "visible on the taste profile page" — but there is no dedicated taste profile route currently.

**How to avoid:** Add a new "Listening History" section to the existing `/settings` page, below the existing Taste Profile / TasteEditor section. Match the existing settings-section card pattern. This avoids new routing, a new nav link, and extra complexity. (Confirmed: Settings page has `aiState.enabled` gating — Listening History section should also be gated on `tauriMode` only, since local playback is Tauri-only.)
