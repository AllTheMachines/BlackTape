Close 3 critical/medium integration gaps that silently break the taste-aware feature set and KB navigation: wire `loadTasteProfile()` at app startup, create a minimum `/about` route, and link genre/scene pages to `/discover`. <decisions>
## Implementation Decisions

### /about page
- Structured stub with real section headings, placeholder text inside (not truly bare minimum)
- Sections to include: What Mercury is / How it works / Why it exists / Mission statement / Data sources (MusicBrainz, etc.)
- Two outbound links: "Start discovering →" CTA pointing to main search, and a GitHub link
- Reachable from: footer link AND main navigation

### Taste profile — no profile (new user)
- AI Explore empty state: minimal stub — a message and link to search ("Tag some artists to get recommendations")
- No popular/trending artist suggestions
- Full onboarding flow is a future phase; this stub is the placeholder

### Taste profile — loading state
- While taste profile is loading at startup: show skeleton loaders (pulsing placeholder cards)
- Not hidden/empty — something visible while profile resolves

### Genre→discover link
- Appearance: inline text link (not a prominent button)
- Placement: both top and bottom of genre/scene pages
- Label format: "Explore [genre] in Discover" (e.g. "Explore ambient in Discover")
- URL param handling: `/discover?tags=<mb_tag>` handles filtering automatically — no extra wiring needed beyond correct URL construction

### Tauri startup sequence
- KB personalized starter graph loads immediately on Tauri launch — not lazy, not background
- Silent load — no "Personalizing..." indicator or startup noise
- If no taste profile: same minimal stub as AI Explore — message + link to search (consistent across both surfaces)
- Settings page taste configuration stays as-is — auto-load removes the blocker but Settings remains the place to manually manage taste

### Claude's Discretion
- Exact skeleton loader design (number, shape) — match existing loading patterns in app
- Specific section text content within /about stubs
- Navigation placement specifics (top nav vs sidebar, link position in footer)
- How to wire `loadTasteProfile()` into `+layout.svelte` `onMount` technically

</decisions>
