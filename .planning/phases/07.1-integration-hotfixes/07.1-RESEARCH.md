## Summary

Phase 07.1 is a wiring/hotfix phase. All the feature code was built in Phases 5–7; none of the underlying functionality needs to be re-implemented. Three integration gaps must be closed: a missing function call in the layout's startup sequence (GAP-01), a missing route (GAP-02), and a missing navigation link (GAP-04). Phase 07.3 (GAP-03, playback→taste signal) is a separate phase.

The root cause of GAP-01, GAP-02, and part of GAP-04 is that features were built in isolation without full end-to-end wiring — the final connection from the shell (layout) and from genre pages into the taste and discovery systems was not made. Each fix is small and surgical.

This phase involves zero new dependencies, zero schema changes, and zero new Tauri commands. All changes are in `.svelte` and `.svelte.ts` files already in the codebase.

**Primary recommendation:** Fix all three gaps in order of impact: GAP-01 (one-line layout fix, unblocks 4 requirements), GAP-02 (/about route creation), GAP-04 (genre page link, completes KB→Discover navigation chain).

## Architecture Patterns

### Pattern 1: GAP-01 — Layout onMount Wiring

**What the code currently does** (lines 19-28 of `+layout.svelte`):
```typescript
onMount(async () => {
    tauriMode = isTauri();

    if (isTauri()) {
        await loadAiSettings();
        if (aiState.enabled) {
            initializeAi();
        }
    }
});
```

**The fix:** `loadTasteProfile()` must be called in the same Tauri-gated block. It uses the same dynamic import pattern (`await import('@tauri-apps/api/core')` inside the function), making it safe to import unconditionally at the module level.

**Correct pattern after fix:**
```typescript
import { loadTasteProfile } from '$lib/taste/profile.svelte';
// ...
onMount(async () => {
    tauriMode = isTauri();

    if (isTauri()) {
        await loadAiSettings();
        loadTasteProfile();   // fire-and-forget — doesn't block AI init
        if (aiState.enabled) {
            initializeAi();
        }
    }
});
```

**Key decision:** `loadTasteProfile()` should NOT be awaited (fire-and-forget). Rationale:
- AI initialization (`initializeAi()`) is already not awaited (it polls for up to 180s)
- Blocking the layout mount on taste profile load would delay the entire app render
- `tasteProfile.isLoaded` flag allows consumers to show skeleton state while loading
- `AiRecommendations` already gates on `tasteProfile.hasEnoughData`, which will become true asynchronously once loading completes

**Source:** Direct code reading — `src/routes/+layout.svelte` lines 1-33, `src/lib/taste/profile.svelte.ts` lines 45-65, `src/lib/ai/state.svelte.ts` lines 104-207.

### Pattern 2: GAP-01 — Skeleton Loading State in KB

**What the KB page currently does** (`src/routes/kb/+page.svelte`):
The `kb/+page.ts` universal load function reads `tasteProfile.tags` synchronously. In Tauri context, `tasteProfile.tags` will be `[]` until `loadTasteProfile()` resolves. After the fix, the load function will run before the profile loads (it runs during route navigation), so the KB page will initially receive a generic graph, then reactivity via `tasteProfile` will not automatically re-run the load function.

**The correct approach for KB personalization:** The KB page (`/kb`) must watch `tasteProfile.isLoaded` reactively and re-fetch/recompute the personalized graph after the profile loads.

Two options:
1. **Reactive re-fetch in `+page.svelte`:** Use `$effect` that watches `tasteProfile.isLoaded` and calls the local DB query when it becomes true — replaces the data without a navigation event
2. **Skeleton state during loading:** Show placeholder graph while `!tasteProfile.isLoaded`, then compute personalized graph once loaded

**Recommended:** Option 1 — reactive re-fetch. The KB page already has the graph rendering logic. An `$effect` that re-queries when `tasteProfile.isLoaded` changes keeps the graph accurate without a full page reload.

```typescript
// In kb/+page.svelte script block (Tauri-only path)
import { tasteProfile } from '$lib/taste/profile.svelte';

let graph = $state(data.graph);  // initial from server/page.ts

$effect(() => {
    if (tasteProfile.isLoaded && isTauri()) {
        // Re-fetch personalized graph once taste profile is ready
        loadPersonalizedGraph();
    }
});

async function loadPersonalizedGraph() {
    try {
        const { getProvider } = await import('$lib/db/provider');
        const { getStarterGenreGraph } = await import('$lib/db/queries');
        const db = await getProvider();
        const tasteTags = tasteProfile.tags.map((t) => t.tag).slice(0, 5);
        graph = await getStarterGenreGraph(db, tasteTags);
    } catch { /* keep current graph */ }
}
```

**Skeleton for KB:** Per user decision, show skeleton loaders while profile loads. The existing `GenreGraph` renders immediately with server data (non-personalized). The skeleton is specifically for the profile-loading state indicator, not blocking graph display. A simple loading indicator below the subtitle suffices.

**Source:** Direct code reading — `src/routes/kb/+page.svelte`, `src/routes/kb/+page.ts`, `src/lib/taste/profile.svelte.ts`.

## Common Pitfalls

### Pitfall 1: Awaiting loadTasteProfile() in onMount
**What goes wrong:** If `loadTasteProfile()` is awaited before `initializeAi()`, app startup blocks for 1-3 seconds waiting for the taste.db read before AI servers even begin starting.
**Why it happens:** Looks natural to await async functions in sequence.
**How to avoid:** Fire-and-forget — call `loadTasteProfile()` without await. The `tasteProfile.isLoaded` flag signals completion to consumers. AI initialization also does not block (it runs in background, polling health).
**Warning signs:** App feels slow to paint on startup; AI indicator doesn't appear immediately.

---

### Pitfall 2: KB Page Personalization Not Reactive After GAP-01 Fix
**What goes wrong:** Even after fixing GAP-01, the KB page loads before the taste profile resolves. The `+page.ts` universal load reads `tasteProfile.tags` synchronously at load time — if the profile isn't loaded yet, it gets `[]`. The profile loading won't re-trigger the page load.
**Why it happens:** SvelteKit's page load functions run once per navigation, not reactively. `tasteProfile` state updates after the load completes.
**How to avoid:** The KB `+page.svelte` must watch `tasteProfile.isLoaded` with a `$effect` and re-query the personalized graph client-side once the profile is ready. This is an additional change beyond the layout fix.
**Warning signs:** KB graph always shows the generic graph even after GAP-01 fix; taste tags in the state but graph is non-personalized.

---

### Pitfall 3: About Page Missing from Tauri Navigation
**What goes wrong:** About page is added to web nav but not Tauri nav (or vice versa). The layout has two separate `<nav>` blocks — one for web, one for Tauri.
**Why it happens:** The layout has two nav blocks that must be updated independently.
**How to avoid:** Update BOTH nav blocks in `+layout.svelte`. The web nav is at line 80 (`{:else}` block) and the Tauri nav is at line 53 (`{#if tauriMode}` block).
**Warning signs:** About link appears on web but not desktop (or vice versa).

---

### Pitfall 4: Genre mb_tag Can Be Null
**What goes wrong:** Not all genres have `mb_tag` set. Constructing `/discover?tags=null` or `/discover?tags=undefined` would produce a broken URL.
**Why it happens:** The genres table allows `mb_tag` to be null for genres that weren't matched to a MusicBrainz tag during pipeline.
**How to avoid:** Wrap the discover link in `{#if data.genre.mb_tag}` — only render the link when a valid tag is available.
**Warning signs:** Discover page with `?tags=null` in URL returns no results.

---

### Pitfall 5: /about Route Needs No Special Config in Tauri
**What goes wrong:** Concern that static adapter won't serve `/about` in Tauri SPA mode.
**Why it happens:** `adapter-static` with `fallback: 'index.html'` is configured for Tauri. Any route not pre-rendered falls back to `index.html`, which then routes via the SvelteKit client-side router. `/about` will be handled correctly.
**How to avoid:** No special config needed. The route just works. Verify by building and navigating to `/about` in dev mode.
**Warning signs:** N/A — this is a pitfall to be aware of, not something to fix.

---

### Pitfall 6: Skeleton in Explore Needs isLoaded Check, Not tags.length Check
**What goes wrong:** Using `tasteProfile.tags.length === 0` to show the skeleton would also show skeleton for new users with no taste data (who have fully loaded profiles).
**Why it happens:** Empty profile and loading profile both result in `tags.length === 0`.
**How to avoid:** Use `tasteProfile.isLoaded` as the condition. Show skeleton when `!tasteProfile.isLoaded`. Show "no data" empty state when `tasteProfile.isLoaded && tasteProfile.tags.length === 0`.
**Warning signs:** New users see a perpetual skeleton instead of the "tag some artists" CTA.
