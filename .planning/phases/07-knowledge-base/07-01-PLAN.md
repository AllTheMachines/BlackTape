  <action>
Append the following table definitions to `pipeline/lib/schema.sql` (after the existing tag_stats and tag_cooccurrence definitions):

```sql
-- Phase G: Genre encyclopedia entities
CREATE TABLE IF NOT EXISTS genres (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  slug TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  type TEXT NOT NULL DEFAULT 'genre',    -- 'genre' | 'scene' | 'city'
  wikidata_id TEXT,
  wikipedia_title TEXT,
  inception_year INTEGER,
  origin_city TEXT,
  origin_lat REAL,
  origin_lng REAL,
  mb_tag TEXT
);

-- Phase G: Genre parent/child + influenced-by relationships
CREATE TABLE IF NOT EXISTS genre_relationships (
  from_id INTEGER NOT NULL REFERENCES genres(id),
  to_id INTEGER NOT NULL REFERENCES genres(id),
  rel_type TEXT NOT NULL DEFAULT 'subgenre',   -- 'subgenre' | 'influenced_by' | 'scene_of'
  PRIMARY KEY (from_id, to_id, rel_type)
);

CREATE INDEX IF NOT EXISTS idx_genre_slug ON genres(slug);
CREATE INDEX IF NOT EXISTS idx_genre_mb_tag ON genres(mb_tag);
CREATE INDEX IF NOT EXISTS idx_genre_rel_from ON genre_relationships(from_id);
CREATE INDEX IF NOT EXISTS idx_genre_rel_to ON genre_relationships(to_id);
```

The `type` column distinguishes three node types for the genre graph visualization (genre = global, scene = geographic/temporal, city = geographic origin node). The `mb_tag` column (normalized lowercase slug) bridges to the existing `artist_tags` table so genre pages can list key artists.

  <name>Task 2: Create build-genre-data.mjs — Wikidata SPARQL fetch, normalization, geocoding, DB insert</name>
  <files>pipeline/build-genre-data.mjs</files>
  <action>
Create `pipeline/build-genre-data.mjs` as an ES module (matching the `.mjs` convention of `build-tag-stats.mjs`). Structure it identically to the existing pipeline scripts — use `better-sqlite3`, progress logging with `console.log('[Phase G]...')`, and idempotent DELETE-before-INSERT.

**Script outline:**

1. **Connect to DB** — same pattern as `build-tag-stats.mjs`: `import Database from 'better-sqlite3'`, open `data/mercury.db`, apply schema.sql via `db.exec(fs.readFileSync(...))`.

2. **fetchWikidataGenres()** — SPARQL query for music genres (wdt:P31 wd:Q188451), their parent genres (wdt:P279), inception year (wdt:P571), country of origin (wdt:P495), and Wikidata influenced-by (wdt:P737). Use LIMIT 5000 and Accept: `application/sparql-results+json`. User-Agent: `Mercury/0.1.0`. Wrap in try/catch — if Wikidata is unreachable, log warning and proceed with empty array (pipeline must not hard-fail).

   ```sparql
   SELECT DISTINCT ?genre ?genreLabel ?genreId ?parentGenre ?parentGenreLabel
                   ?inceptionYear ?originLabel ?influencedBy ?influencedByLabel WHERE {
     ?genre wdt:P31 wd:Q188451 .
     BIND(STRAFTER(STR(?genre), "entity/") AS ?genreId)
     OPTIONAL { ?genre wdt:P279 ?parentGenre . }
     OPTIONAL { ?genre wdt:P571 ?inception . BIND(YEAR(?inception) AS ?inceptionYear) }
     OPTIONAL {
       ?genre wdt:P495 ?origin .
       ?origin rdfs:label ?originLabel .
       FILTER(LANG(?originLabel) = "en")
     }
     OPTIONAL { ?genre wdt:P737 ?influencedBy . }
     SERVICE wikibase:label { bd:serviceParam wikibase:language "en". }
   }
   LIMIT 5000
   ```

3. **slugify(name)** — `name.toLowerCase().replace(/[^a-z0-9]+/g, '-').replace(/^-|-$/g, '')`. For collisions: append first 8 chars of Wikidata Q-number: `slug + '-' + qdataId.slice(1, 9)`.

4. **insertGenres(bindings)** — Loop through SPARQL result bindings. For each unique genre:
   - Determine `type`: if `originLabel` exists and differs from country name → `'scene'`, else `'genre'`
   - `mb_tag` = slugified lowercase genre name (bridge to artist_tags)
   - `wikipedia_title` = `genreLabel.value` (for page/summary fetch at runtime)
   - INSERT OR IGNORE into genres (slug, name, type, wikidata_id, wikipedia_title, inception_year, origin_city, mb_tag)
   - Collect parent/influenced-by pairs for relationship table

5. **insertRelationships(pairs)** — For each parent-child pair from SPARQL: look up both genre IDs, INSERT OR IGNORE into genre_relationships (from_id, to_id, rel_type='subgenre'). For influenced-by pairs: rel_type='influenced_by'. Skip pairs where either ID is missing.

6. **geocodeScenes()** — Query `SELECT id, origin_city FROM genres WHERE type='scene' AND origin_city IS NOT NULL AND origin_lat IS NULL`. For each: call Nominatim `https://nominatim.openstreetmap.org/search?q={encodedCity}&format=json&limit=1&featuretype=city` with User-Agent `Mercury/0.1.0`. Wait 1100ms between requests. On success: UPDATE genres SET origin_lat=?, origin_lng=? WHERE id=?. Log progress every 10 geocoded cities.

7. **main()** — Run steps in order, log counts at each step:
   ```
   [Phase G] Fetching genre data from Wikidata...
   [Phase G] Inserted N genres (M relationships)
   [Phase G] Geocoding K scene cities via Nominatim...
   [Phase G] Genre data build complete.
   ```
   Run after everything else (this is a standalone script, not integrated into import.js main loop — it's run separately: `node pipeline/build-genre-data.mjs`).

**Key constraints from research:**
- NEVER call Nominatim at runtime — pipeline only
- 1100ms delay between Nominatim requests (1 req/sec limit)
- LIMIT 5000 on SPARQL to avoid timeout
- Use `INSERT OR IGNORE` for idempotency
- `DELETE FROM genre_relationships; DELETE FROM genres;` before inserting (idempotent re-run)
- If Wikidata returns 0 results (network issue), log warning and exit 0 — do not crash
