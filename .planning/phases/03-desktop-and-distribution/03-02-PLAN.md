    **Step 1: Install npm dependencies**
    ```
    npm install @tauri-apps/api @tauri-apps/plugin-sql @tauri-apps/plugin-updater @tauri-apps/plugin-process @sveltejs/adapter-static cross-env
    npm install -D @tauri-apps/cli
    ```

    **Step 2: Initialize Tauri project**
    Run `npx tauri init` with these answers:
    - App name: Mercury
    - Window title: Mercury
    - Frontend dev URL: http://localhost:5173
    - Frontend dist: ../build
    - Frontend dev command: npm run dev
    - Frontend build command: cross-env TAURI_ENV=1 npm run build

    If `tauri init` is interactive and can't be scripted, manually create the files instead.

    **Step 3: Configure src-tauri/Cargo.toml**
    Add dependencies:
    ```toml
    [dependencies]
    tauri = { version = "2", features = [] }
    tauri-plugin-sql = { version = "2", features = ["sqlite"] }
    tauri-plugin-updater = "2"
    tauri-plugin-process = "2"
    serde = { version = "1", features = ["derive"] }
    serde_json = "1"
    ```

    **Step 4: Configure src-tauri/src/lib.rs**
    Register plugins:
    ```rust
    pub fn run() {
        tauri::Builder::default()
            .plugin(tauri_plugin_sql::Builder::new().build())
            .plugin(tauri_plugin_updater::Builder::new().build())
            .plugin(tauri_plugin_process::init())
            .run(tauri::generate_context!())
            .expect("error while running tauri application");
    }
    ```

    **Step 5: Configure src-tauri/src/main.rs**
    ```rust
    #![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]

    fn main() {
        mercury_lib::run();
    }
    ```
    (Use the actual lib name from Cargo.toml -- likely `mercury` or `mercury_lib`.)

    **Step 6: Configure src-tauri/tauri.conf.json**
    Key settings:
    - `productName`: "Mercury"
    - `identifier`: "com.mercury.app"
    - `build.beforeDevCommand`: "npm run dev"
    - `build.beforeBuildCommand`: "cross-env TAURI_ENV=1 npm run build"
    - `build.devUrl`: "http://localhost:5173"
    - `build.frontendDist`: "../build"
    - Window: 1200x800, min 800x600, title "Mercury"
    - `app.security.csp`: null (allow external API calls to MusicBrainz, Wikipedia, etc.)

    **Step 7: Configure src-tauri/capabilities/default.json**
    Grant permissions for SQL plugin, updater, process, and HTTP access:
    ```json
    {
      "$schema": "../gen/schemas/desktop-schema.json",
      "identifier": "default",
      "description": "Default capability for Mercury desktop",
      "windows": ["main"],
      "permissions": [
        "core:default",
        "sql:default",
        "sql:allow-select",
        "sql:allow-execute",
        "updater:default",
        "process:default"
      ]
    }
    ```

    **Step 8: Update svelte.config.js for conditional adapter**
    ```javascript
    import adapterCloudflare from '@sveltejs/adapter-cloudflare';
    import adapterStatic from '@sveltejs/adapter-static';

    const isDesktop = process.env.TAURI_ENV === '1';

    const config = {
      kit: {
        adapter: isDesktop
          ? adapterStatic({ fallback: 'index.html' })
          : adapterCloudflare({
              routes: { include: ['/*'], exclude: ['<all>'] }
            })
      }
    };

    export default config;
    ```

    **Step 9: Update vite.config.ts for desktop build**
    Add `build.assetsInlineLimit: 0` when building for desktop (prevents CSP issues in Tauri webview):
    ```typescript
    import { sveltekit } from '@sveltejs/kit/vite';
    import { defineConfig } from 'vite';

    const isDesktop = process.env.TAURI_ENV === '1';

    export default defineConfig({
      plugins: [sveltekit()],
      ...(isDesktop ? {
        build: {
          assetsInlineLimit: 0
        }
      } : {})
    });
    ```

    **Step 10: Create src/routes/+layout.ts**
    This file configures SvelteKit for SPA mode in the desktop build. When using adapter-static with a fallback, SSR must be disabled:
    ```typescript
    export const ssr = false;
    export const prerender = false;
    ```
    IMPORTANT: This file only takes effect in the desktop build (adapter-static). In the web build (adapter-cloudflare), the +layout.server.ts (if any) and SSR work normally because the Cloudflare adapter ignores these settings. However, since there is no existing +layout.server.ts, and the web build does use SSR via +page.server.ts files, we need to make this conditional. The cleanest approach: only create this file and set `ssr = false` when building for desktop. Use Vite's `define` to inject a build-time constant, or simply accept that the web build also sets `ssr = false` at the layout level (the +page.server.ts files still run because SvelteKit server load functions execute regardless of the ssr setting -- they run on the server during SSR AND during client-side navigation). Test this: run `npm run build` (web) and confirm +page.server.ts routes still work.

    Actually, the safer approach: always set `ssr = false` and `prerender = false` in this file. For the Cloudflare adapter, the +page.server.ts load functions still execute on the server for initial page loads, and on the server for client navigation (SvelteKit calls them via internal fetch). The `ssr` layout option only controls whether the HTML is server-rendered -- the data loading still happens server-side. So this is safe for both builds.
