<objective>
Create a database abstraction layer so the same query functions work with both Cloudflare D1 (web) and Tauri SQL plugin (desktop).

Purpose: The existing codebase hardcodes `D1Database` everywhere. Tauri has no D1 -- it uses `@tauri-apps/plugin-sql` with a different API. This plan introduces a `DbProvider` interface that both D1 and Tauri implement, then refactors all query functions and route handlers to use it. After this plan, the web build works exactly as before, and the desktop build has a provider ready to use.

Output: Provider interface, two implementations (D1 + Tauri), refactored queries.ts, updated route handlers.
</objective>

  <action>
    Create three files:

    **src/lib/db/provider.ts:**
    - Define `DbProvider` interface with two methods:
      - `all<T>(sql: string, ...params: unknown[]): Promise<T[]>` -- returns all matching rows
      - `get<T>(sql: string, ...params: unknown[]): Promise<T | null>` -- returns first row or null
    - Export a `getProvider()` factory function that detects the platform and returns the right provider:
      - Check `isTauri` from `@tauri-apps/api/core` (wrap in try/catch for when the package isn't installed yet)
      - If Tauri: return a TauriProvider singleton
      - If not Tauri: throw an error (D1Provider is created explicitly from platform.env.DB in server routes, not via factory)
    - The factory is async because TauriProvider needs to call `Database.load()`.

    **src/lib/db/d1-provider.ts:**
    - Class `D1Provider` implements `DbProvider`.
    - Constructor takes `D1Database`.
    - `all<T>()`: calls `this.db.prepare(sql).bind(...params).all<T>()`, returns `results`.
    - `get<T>()`: calls `all<T>()`, returns `results[0] ?? null`.
    - Keep it simple -- this is a thin wrapper.

    **src/lib/db/tauri-provider.ts:**
    - Class `TauriProvider` implements `DbProvider`.
    - Uses lazy singleton pattern: `Database.load('sqlite:mercury.db')` on first use.
    - `all<T>()`: calls `db.select<T[]>(sql, params)`. Note: tauri-plugin-sql takes params as an array, not spread.
    - `get<T>()`: calls `all<T>()`, returns `results[0] ?? null`.
    - Import `Database` from `@tauri-apps/plugin-sql`.
    - The import will fail in the web build -- that's fine because this file is only imported in the Tauri code path (tree-shaken out of web build, or dynamically imported).
    - Use dynamic import `await import('@tauri-apps/plugin-sql')` inside the load function to avoid build-time failures in the web context.
  </action>
  <verify>
    Run `npm run check` -- TypeScript should compile without errors (the Tauri imports will be type-checked once the packages are installed in Plan 02, but provider.ts and d1-provider.ts should be clean now). Verify the interface is exported correctly.
  </verify>
  <done>
    Three files exist. DbProvider interface has `all` and `get` methods. D1Provider wraps D1Database. TauriProvider uses dynamic import of @tauri-apps/plugin-sql. The web build is not broken.
