<objective>
Handle the first-run experience when no local database exists, and create the database distribution pipeline.

Purpose: The 816MB mercury.db is NOT bundled with the installer -- it's too large. On first launch, the desktop app needs to detect that the database is missing and guide the user to obtain it. This plan creates: (1) a Tauri command that checks for the database file, (2) a first-run UI component, (3) a compression script for distribution. The initial approach is simple: the user downloads the compressed database file and the app detects it. In-app download with progress bar is a future enhancement.

Output: First-run detection, setup UI, database compression pipeline.
</objective>

    **Part A: Tauri Rust command for database detection**

    Add a Tauri command to `src-tauri/src/lib.rs` that:
    1. Resolves the app data directory path (using `tauri::Manager` and `app.path()`)
    2. Checks if `mercury.db` exists in that directory
    3. Returns a JSON object: `{ exists: bool, path: String, dir: String }`

    ```rust
    #[tauri::command]
    fn check_database(app: tauri::AppHandle) -> Result<serde_json::Value, String> {
        let app_data = app.path().app_data_dir().map_err(|e| e.to_string())?;
        let db_path = app_data.join("mercury.db");
        Ok(serde_json::json!({
            "exists": db_path.exists(),
            "path": db_path.to_string_lossy(),
            "dir": app_data.to_string_lossy()
        }))
    }
    ```

    Register the command in the Builder:
    ```rust
    tauri::Builder::default()
        .invoke_handler(tauri::generate_handler![check_database])
        .plugin(...)
        // ... rest of plugins
    ```

    **Part B: Update TauriProvider to use correct database path**

    Modify `src/lib/db/provider.ts` (or `tauri-provider.ts`) so the Database.load() call uses the correct app data directory path. The tauri-plugin-sql `Database.load('sqlite:path')` needs the full path to the database file in the app data directory.

    Update the TauriProvider to accept a path parameter or resolve it via Tauri's path API:
    ```typescript
    // In tauri-provider.ts, update the getDb function:
    async function getDb(): Promise<Database> {
      if (!dbInstance) {
        // Use Tauri's path API to get the app data directory
        const { appDataDir } = await import('@tauri-apps/api/path');
        const dir = await appDataDir();
        dbInstance = await Database.load(`sqlite:${dir}mercury.db`);
      }
      return dbInstance;
    }
    ```

    **Part D: Wire into landing page**

    Modify `src/routes/+page.svelte` to:
    1. On mount, if `isTauri()` is true, call `invoke('check_database')` to check if DB exists
    2. If DB doesn't exist, show the DatabaseSetup component instead of the normal search UI
    3. If DB exists (or if not Tauri), show the normal landing page
    4. The "Check Again" button re-runs the check and updates the state

    Use `$state` for the database status. The check runs on mount via `$effect` or `onMount`.

    ```svelte
    <script lang="ts">
      import { isTauri } from '$lib/platform';
      import DatabaseSetup from '$lib/components/DatabaseSetup.svelte';
      // ... existing imports

      let dbStatus = $state<'checking' | 'ready' | 'missing'>('checking');
      let dbPath = $state('');

      async function checkDatabase() {
        if (!isTauri()) {
          dbStatus = 'ready';
          return;
        }
        const { invoke } = await import('@tauri-apps/api/core');
        const result = await invoke('check_database');
        dbStatus = result.exists ? 'ready' : 'missing';
        dbPath = result.path;
      }

      $effect(() => { checkDatabase(); });
    </script>

    {#if dbStatus === 'checking'}
      <!-- Brief loading state -->
    {:else if dbStatus === 'missing'}
      <DatabaseSetup {dbPath} onRetry={checkDatabase} />
    {:else}
      <!-- Existing landing page content -->
    {/if}
    ```

<task type="auto">
  <name>Task 2: Database compression pipeline</name>
  <files>
    pipeline/compress-db.js
  </files>
  <action>
    Create a Node.js script at `pipeline/compress-db.js` that:

    1. Reads `pipeline/data/mercury.db` (the 816MB source database)
    2. Runs SQLite `VACUUM` to optimize before compression (use better-sqlite3 or shell out to sqlite3 CLI)
    3. Compresses with gzip (built-in Node.js zlib) to produce `pipeline/data/mercury.db.gz`
       - Note: The research suggests zstd, but zstd requires a native module. Start with gzip which is built into Node.js. The compression ratio will be somewhat worse (~120-160MB instead of ~80-120MB with zstd), but it works everywhere without dependencies. Switch to zstd later if size matters.
    4. Reports: original size, compressed size, compression ratio
    5. Optionally create a SHA256 checksum file (`mercury.db.gz.sha256`) for integrity verification

    Usage: `node pipeline/compress-db.js`

    The script should:
    - Check that the source file exists before proceeding
    - Show progress (e.g., "Compressing... this may take a minute")
    - Use streaming (createReadStream -> gzip -> createWriteStream) to avoid loading 816MB into memory
    - Print final sizes and ratio when done

    Do NOT use `better-sqlite3` for VACUUM -- it's a native module that may not be installed. Instead, either:
    - Skip VACUUM (the pipeline already produces a clean DB)
    - Or shell out to `sqlite3` CLI if available: `sqlite3 mercury.db "VACUUM;"`
    - The simplest approach: just compress without VACUUM. Add VACUUM as a future optimization.
  </action>
  <verify>
    1. If `pipeline/data/mercury.db` exists, run `node pipeline/compress-db.js` and confirm it produces `pipeline/data/mercury.db.gz`.
    2. If the source DB doesn't exist, the script should print a clear error message and exit.
    3. The compressed file should be significantly smaller than 816MB.
  </verify>
  <done>
    Compression script exists and works. Running it produces a distributable .gz file from the source database. SHA256 checksum is generated for integrity verification.
  </done>
