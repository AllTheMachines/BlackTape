## Architecture Patterns

### Pattern 1: Graceful Platform Guard (Existing Convention)

**What:** Check `platform?.env?.DB` with optional chaining and return a safe default when DB is unavailable, instead of crashing with `platform!.env.DB` non-null assertion.

**When to use:** Every server-side route that accesses D1.

**Existing examples (HIGH confidence -- from the codebase):**

```typescript
// Pattern A: PageServerLoad returning empty data
// Source: src/routes/kb/+page.server.ts
export const load: PageServerLoad = async ({ platform }) => {
    if (!platform?.env?.DB) {
        return { graph: { nodes: [], edges: [] } };
    }
    const db = new D1Provider(platform.env.DB);
    // ... query and return data
};

// Pattern B: API route returning JSON error
// Source: src/routes/api/search/+server.ts
export const GET: RequestHandler = async ({ url, platform }) => {
    const db = platform?.env?.DB;
    if (!db) {
        return json({ error: 'Search unavailable — database not connected' }, { status: 503 });
    }
    // ... query and return data
};
```

**Four files that need this pattern applied:**

| File | Current Line | Return Shape on Guard |
|------|-------------|----------------------|
| `src/routes/time-machine/+page.server.ts:9` | `new D1Provider(platform!.env.DB)` | `{ artists: [], year }` |
| `src/routes/api/time-machine/+server.ts:14` | `new D1Provider(platform!.env.DB)` | `json({ artists: [], year }, { status: 503 })` |
| `src/routes/api/genres/+server.ts:7` | `new D1Provider(platform!.env.DB)` | `json({ nodes: [], edges: [] })` or 503 |
| `src/routes/kb/genre/[slug]/+page.server.ts:43` | `new D1Provider(platform!.env.DB)` | `error(503, 'Database unavailable')` or return empty genre shell |

**Design decision for genre detail page:** The genre detail route (`kb/genre/[slug]/+page.server.ts`) is more complex than the others because it has a 404 check (`if (!genre) throw error(404, 'Genre not found')`). When DB is unavailable, it should return an empty shell or throw a 503, not a 404. The PageServerLoad pattern (return empty data) is preferred over throwing error(503) because Tauri SPA mode uses SSR=false and these server loads don't execute in Tauri context anyway -- the guard is for test environments without D1.

**What:** SQL comments in `schema.sql` documenting which pipeline scripts create which tables and in what order.

**Current state:** `schema.sql` defines `artists`, `artist_tags`, `artists_fts`, `genres`, `genre_relationships` tables. But `tag_stats` and `tag_cooccurrence` are created only by `pipeline/build-tag-stats.mjs` at runtime. The file has no comments about execution order.

2. `pipeline/build-tag-stats.mjs` -- Phase F: tag_stats + tag_cooccurrence tables

### Anti-Patterns to Avoid

- **Partial checkbox updates:** When updating REQUIREMENTS.md, update BOTH the checkbox section AND the traceability table. The audit found cases where one was updated but not the other.
- **Inconsistent frontmatter:** Phase 5 VERIFICATION.md has `status: passed` in frontmatter but `status: gaps_found` in the body. When the gap was resolved, the body text wasn't updated to match. Don't introduce new inconsistencies.
- **Over-engineering guards:** The `platform!.env.DB` fix is a simple guard pattern. Don't add retry logic, error boundaries, or middleware. Match the existing convention exactly.

## Common Pitfalls

### Pitfall 1: REQUIREMENTS.md Checkbox vs Traceability Table Desynchronization

**What goes wrong:** Updating the checkbox at the top of REQUIREMENTS.md but forgetting to update the matching row in the Traceability table (or vice versa).
**Why it happens:** The two sections are far apart in the file (checkboxes at lines 30-35, traceability at lines 136-196).
**How to avoid:** For each requirement update, make TWO edits: one in the checkbox section, one in the traceability table. Verify both before committing.
**Warning signs:** `grep "PLAYER-01" .planning/REQUIREMENTS.md` should show consistent status in both locations.

### Pitfall 2: Retroactive VERIFICATION.md Claiming Passed Tests That Were Never Run

**What goes wrong:** Marking runtime tests as "passed" when they were only compile-checked.
**Why it happens:** The 04-UAT.md shows all 11 tests as "skipped" but the human verification in 04-05-SUMMARY.md confirmed them manually.
**How to avoid:** Clearly distinguish: (1) compile-time verified artifacts (npm check, cargo check, file existence), (2) human-verified at Phase 4 completion (2026-02-17), (3) runtime tests that remain skipped (no headless path). The verification report should note that human verification was done per 04-05-SUMMARY.md but formal GSD verification was not.
**Warning signs:** A score like "11/11 passed" when 11 runtime tests were skipped.

### Pitfall 3: Genre Detail Page Guard Breaking the 404 Path

**What goes wrong:** The genre detail `+page.server.ts` has `if (!genre) throw error(404, ...)`. If the guard returns empty data instead of throwing, the page renders with null genre data instead of showing 404.
**Why it happens:** The guard fires before the genre lookup, so there's no genre to check.
**How to avoid:** When DB is unavailable, return a data shape that the page component can gracefully handle (e.g., `{ genre: null, keyArtists: [], subgraph: { nodes: [], edges: [] }, wikipediaSummary: null }`) and let the page component show an appropriate message. OR throw error(503) which is more correct for "database unavailable".
**Warning signs:** Genre pages showing blank content instead of 404 for non-existent genres, or 503 pages showing up in Tauri dev mode.

### Pitfall 4: Phase 5 VERIFICATION.md Frontmatter Fix Going Out of Scope

**What goes wrong:** The v1.0 audit notes that Phase 5 VERIFICATION.md has inconsistent frontmatter (`status: passed` but body says `gaps_found`). This is tempting to fix but is NOT in the Phase 07.3 success criteria.
**Why it happens:** The audit lists it as tech debt but Phase 07.3's scope is specifically PLAYER-01/02/03 and AI-03 documentation.
**How to avoid:** Stick to the 6 success criteria in the ROADMAP. If Phase 5 VERIFICATION.md is encountered, note it but don't modify it unless it's part of a broader traceability update.

### Pitfall 5: REQUIREMENTS.md AI-03 Already Updated

**What goes wrong:** Attempting to update AI-03 when it has already been marked `[x]` Complete in a prior session.
**Why it happens:** The REQUIREMENTS.md was updated during Phase 07.1/07.2 gap closure work.
**How to avoid:** Read the current state of REQUIREMENTS.md before making changes. As of the last read (this research session), AI-03 line 41 already shows `[x]` with the note "artist summaries (Phase 5) + genre/scene summaries (Phase 7) both complete" and the traceability row at line 151 shows "Complete". **No AI-03 update may be needed.** Verify at plan time.

## Code Examples

### Graceful Platform Guard for PageServerLoad

```typescript
// Source: Existing pattern in src/routes/kb/+page.server.ts (verified)
// Apply to: src/routes/time-machine/+page.server.ts

import type { PageServerLoad } from './$types';
import { D1Provider } from '$lib/db/d1-provider';
import { getArtistsByYear } from '$lib/db/queries';

const DEFAULT_YEAR = new Date().getFullYear() - 30;

export const load: PageServerLoad = async ({ url, platform }) => {
    const year = parseInt(url.searchParams.get('year') ?? String(DEFAULT_YEAR), 10);

    if (!platform?.env?.DB) {
        return { artists: [], year };
    }

    const db = new D1Provider(platform.env.DB);
    const artists = await getArtistsByYear(db, year, undefined, 50);
    return { artists, year };
};
```
